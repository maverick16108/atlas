

Техническое задание на разработку
аукционной платформы для торговли золотом

1. Общая Информация

Назначение системы:

Разработка специализированной автоматизированной аукционной платформы для осуществления торгов золотом и другими драгоценными металлами среди предварительно аккредитованных участников.

2. Функциональные Требования

2.1 Управление Пользователями

●	Регистрация/авторизация пользователей с отметкой о наличии аккредитации.
●	Создание учетных записей для существующих контрагентов (около 28 организаций).
●	Возможность настройки прав доступа и ограничений для отдельных категорий пользователей.
●	Двухфакторная аутентификация (по желанию заказчика).

2.2 Кабинет Пользователя

●	Базовые сведения о пользователе (логистика, базис поставки и др.)
●	Настройка коммерческих условий (логистика, базы поставок)
●	Ограничения по количеству приобретаемого металла (пробные лоты, минимальные объемы закупок)

2.3 Информирование и Оповещение Участников

●	Рассылка уведомлений о начале торгов, изменении цен, окончании торгов.
●	Уведомления рассылаются по различным сценариям, с учетом наличия заключенных договоров с производителями сырья, отметок о аккредитации и прочих признаков

2.4 Публикация Лоттов

●	Поддержка различных типов драгоценных металлов (золото, серебро и др.)
●	Карточка товара (размер слитков, марка, производитель)
●	Указание количества единиц товара в каждом лоте
●	Возможности фильтрации лотов по характеристикам (производители, тип металла)

2.5 Сбор Котировок

●	Механизм подачи заявок участниками торгов.
●	Возможность установки минимальной начальной цены.
●	Алгоритм автоматического повышения ставок (шаг увеличения).
●	Учёт особенностей (право первой руки Газпромбанка, лимиты на закупку).
●	Гибкость настроек (ограничения по объему закупки, разделение партий).

2.6 Формирование Итогов Торгов

●	Определение победителя торгов по лучшим условиям.
●	Генерация протоколов торгов и пакета документации для заключения сделки.
●	Электронная цифровая подпись (на данном этапе не рассматриваем).

2.7 Отчетность

●	Анализ исторических данных (отчет с накопительным итогом по году).
●	Предоставление отчетности по различным сценариям сделок (НДС / Без НДС, внутренняя торговля / Экспорт).
●	Мониторинг активности участников торгов.
●	Интерфейсы API для интеграции с внешними системами.

3. Нефункциональные Требования

3.1 Безопасность и Конфиденциальность

●	Шифрование передаваемых данных.
●	Регулярные резервные копии базы данных.
●	Соответствие законодательству РФ в сфере персональных данных и коммерческой тайны.

3.2 Производительность и Нагрузка

●	Поддержка одновременного подключения до 50 пользователей.
●	Время отклика интерфейса не более 2 секунд.
●	Устойчивость к пиковым нагрузкам (до 100 операций в минуту).

3.3 Удобство Использования

●	Простой и интуитивно понятный интерфейс.
●	Адаптация под мобильные устройства.
●	Минималистичный дизайн, удобный для восприятия финансовых данных.

3.4 Масштабируемость

●	Возможность расширения функционала в будущем.
●	Добавление новых модулей и интеграций.
